# 개인정보 처리 아키텍처 (영상 비저장 · ISMS-P 대응 슬라이드)

> 목적: 카메라 인원 카운트의 **개인정보보호법 제25조(영상정보처리기기)** 리스크를 선제 차단.
> 핵심 메시지: **"영상은 저장·전송하지 않는다. 네트워크로 나가는 건 사람 수(정수)뿐이다."**

---

## 1. 데이터 흐름 (엣지 추론 → 숫자만 전송)

```
 [노트북 웹캠]
      │  RGB 프레임 (메모리 내 휘발)
      ▼
 ┌─────────────────────────────┐
 │  엣지 추론 (camera/counter.py) │   ← 영상은 이 박스 밖으로 안 나감
 │  YOLO person 검출 → N명 카운트  │
 │  추론 직후 프레임 즉시 폐기(del) │
 └─────────────────────────────┘
      │  occupancy=N  (정수 1개. 영상·좌표·얼굴 없음)
      ▼
 POST /api/v1/sensor/reading {space_id, device_id:"cam-laptop", occupancy:N}
      ▼
 [백엔드]  Wells-Riley 재실인원 n 으로만 사용 → tier 산출
      ▼
 [DB]  occupancy 컬럼 = 집계 인원수만 저장 (개인 식별 불가)
```

## 2. 개인정보보호 설계 원칙 (체크리스트)

| 원칙 | 구현 | 근거 |
|---|---|---|
| 영상 **비저장** | 프레임 추론 후 `del frame` — 디스크·DB 기록 없음 | 개인정보보호법 §25, ISMS-P |
| 영상 **비전송** | 네트워크 페이로드 = `occupancy` 정수뿐 | 최소수집 원칙 §3-1 |
| **개인 식별 불가** | 얼굴인식·재식별·추적 미수행, 단순 카운트 | 가명·익명처리 |
| **엣지 처리** | 추론을 단말(노트북)에서 수행, 원본 미반출 | 데이터 국지화 |
| **목적 한정** | 감염위험(재실밀도) 산정 단일 목적 | 목적구속 원칙 |

## 3. 대안 센서 (프라이버시 강화 옵션)

영상 자체가 부담스러운 민감구역(병실 내부)은 **비영상 레이더로 대체** 가능:

| 방식 | 인원수 정확도 | 프라이버시 | 비고 |
|---|---|---|---|
| 카메라+YOLO (현재) | 높음 | 영상 비저장 설계로 완화 | 데모·복도·공용부 |
| **HLK-LD2450** 24GHz 레이더 | 최대 3명 추적 | ✅ 비영상(원천적) | 병실 내부 권장, ~$5 |
| LD2410C (현재 라파이) | 유무(1/0)만 | ✅ 비영상 | 백업·게이팅 |

→ 발표 카드: **"공용부=카메라(정확도), 병실=레이더(프라이버시)"** 하이브리드 배치로 법적·실용 균형.

## 4. QA 방어

- **Q. CCTV 설치 아닌가? 동의·안내판 의무는?** → "영상을 저장/전송하지 않는 **엣지 카운터**라 영상정보처리기기 운영과 구분됩니다. 다만 운영 시 안내·운영방침은 보수적으로 준수하고, 병실 등 민감구역은 비영상 레이더로 설계합니다."
- **Q. 숫자도 개인정보인가?** → "집계 인원수는 특정 개인을 식별하지 않습니다. 개인 식별·행동 프로파일링을 수행하지 않습니다."
