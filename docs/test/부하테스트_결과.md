# 다병동 부하테스트 결과 (조달 SW 품질목표 p95<500ms)

> 측정: 2026-06-08 · 대상: `POST /api/v1/sensor/reading` (실DB 적재 + tier 계산 전체 경로)
> 환경: GCP e2-standard-2(2 vCPU) **공용 dev 박스** — UIS DB·Kafka·Qdrant·Redis 동거
> 도구: `tools/loadtest.py` (asyncio + httpx, 5병동 라운드로빈)

## 핵심 결과

| 부하(동시) | 처리량 | p50 | p95 | p99 | 에러 | 판정 |
|---|---|---|---|---|---|---|
| **5 (실 시연 대표)** | 188 rps | 19ms | **60ms** | 90ms | 0 | **PASS ✅** |
| 10 | 60 rps | 82ms | 523ms | 1313ms | 0 | 경계 |
| 20 | 46 rps | 147ms | 2041ms | 4600ms | 0 | 단일인스턴스 한계 |

- **실 시연 부하(5병동 × ~2초 주기 = 집계 동시 ~2~5)에서 p95 60ms — 목표 500ms의 1/8**, 에러 0.
- 다병동 적재 **균등 분산 검증**: 5개 물리 병동에 각 1210건 정확히 분배(`_resolve_space` 해시/인덱스 매핑).

## 발견·수정한 아키텍처 결함 (부하테스트가 잡음)

### 1. 코웨이 클라우드 호출 cache-stampede (치명적)
- **증상**: 초기 측정 p95 **9757ms**, 처리량 20rps.
- **원인**: ingest 핫패스의 `_coway_aq()`가 코웨이 IoCare 클라우드를 **동기 호출**.
  5초 캐시 만료 순간 동시 요청 50개가 한꺼번에 느린 클라우드를 때림(stampede).
- **수정**: 캐시 즉시 반환 + 만료 시 **백그라운드 단일 갱신**(`asyncio.create_task`)으로 전환.
  센서 수신 지연이 가전 클라우드 응답속도에 더 이상 종속되지 않음.
- **효과**: p95 9757ms → 1113ms (8.7배 개선), 에러 14 → 0.

### 2. DB 커넥션 풀 부족
- `asyncpg` 풀 `max_size=4` → 동시 다수 요청이 커넥션 큐잉.
- **수정**: `min_size=4, max_size=12` (2 vCPU 환경 적정값. 20은 컨텍스트 경합으로 오히려 악화).

## 한계 & 상용화 스케일링 경로 (정직 평가)
- **고동시(20+)에서 테일 지연 악화**: 2 vCPU 공용 박스(UIS·Kafka·Qdrant 동거)의 CPU/이벤트루프 포화.
  단일 uvicorn 워커 한계 → **운영은 ① 전용 인스턴스 ② uvicorn `--workers` 다중화
  ③ k8s HPA 수평확장**으로 해소(상태 비저장 ingest라 수평확장 친화적).
- 실 요양병원 규모(수십 병동, 병동당 수초 주기)는 집계 수 rps 수준 → 단일 인스턴스로 충분,
  대규모/다기관 동시 운영 시 위 수평확장 적용.

## 재현
